# 应用前景与安全价值

## 1. 安全场景定位

本项目面向 AI 编译基础设施安全。MLIR/LLVM 已经广泛用于 AI 编译、异构计算、GPU 编译和模型部署链路。上层框架通常会将模型或算子 lowering 到 MLIR，再通过多条 pass pipeline 做优化、bufferization、dialect conversion 和目标代码生成。

如果底层编译 pass 被特定 IR 触发崩溃、断言失败或异常分歧，影响面可能包括：

- 云端模型编译服务被构造输入触发拒绝服务。
- 自动化构建和 CI/CD 流水线中断。
- AI 框架升级 LLVM/MLIR 后出现隐藏兼容性风险。
- 第三方模型、算子或编译插件引入供应链不稳定因素。
- 大规模编译任务中难以人工定位 pass 组合问题。

因此，本项目不是传统意义上的 Web 漏洞扫描器，而是面向 AI 编译供应链和编译基础设施鲁棒性的安全检测工具。

## 2. 应用前景

### 2.1 AI 编译供应链安全测试

AI 框架、模型编译器和推理引擎通常依赖复杂的编译后端。本系统可以作为工具链升级前的回归测试组件，对不同版本 `mlir-opt`、不同 pass pipeline 和不同 dialect seed 进行批量验证，帮助发现供应链中的不稳定点。

适用对象：

- AI 编译器开发团队。
- 模型部署平台。
- 使用 MLIR/LLVM 作为后端的高校和科研项目。
- 需要验证编译工具链安全性的企业研发团队。

### 2.2 云端编译服务拒绝服务风险检测

云端编译服务会接收用户模型、算子或 IR。如果攻击者可以构造触发编译器 hard failure 的输入，就可能造成任务中断或服务资源浪费。本系统可以在上线前批量筛查这类触发条件，并通过最小化复现帮助开发者修复问题。

### 2.3 国产 AI 编译生态适配测试

今年比赛要求中建议结合实际需求并推广国产密码技术、国产软硬件。本项目可扩展到国产 AI 芯片、国产编译工具链和国产操作系统环境中的 MLIR/LLVM 兼容性测试：

- 在国产 CPU/操作系统环境中运行同一 seed suite。
- 针对国产 AI 芯片后端增加专用 dialect 和 lowering pipeline。
- 对国产编译工具链升级做版本差分。
- 将硬件相关 lowering failure 纳入安全鲁棒性评估。

当前项目已经具备跨环境 runner 和版本差分框架，后续只需增加对应工具链配置和 seed 即可扩展。

### 2.4 上游漏洞反馈和开源生态贡献

MIN-004 这类最小化复现可以整理成上游 issue 报告。相比直接提交大段原始 IR，最小复现更容易被维护者接受，也更利于定位根因。项目可以持续运行在新的 LLVM release 或 main 分支上，形成对开源编译基础设施的持续反馈机制。

## 3. 安全价值

| 能力 | 安全价值 |
| --- | --- |
| 真实 seed 导入 | 用真实生态输入减少玩具样例偏差。 |
| 多 pipeline 差分 | 发现单 pass 测试难以覆盖的组合风险。 |
| failure clustering | 从大量日志中快速合并同类问题，降低研判成本。 |
| P0/P1/P2 triage | 优先处理 hard failure 和当前版本候选。 |
| seed minimization | 将复杂输入压缩为可修复、可上报、可答辩的证据。 |
| 版本差分 | 区分历史遗留、已修复、当前版本和版本兼容问题。 |
| Web 展示台 | 支持现场演示、结果复核和源码检查。 |

## 4. 与命题赛方向的关联

虽然本项目选择自由作品赛道，但它与命题挑战赛中的“APT 供应链攻击检测与溯源系统”和“基于大语言模型的应用安全审计技术”存在应用交叉：

- 从供应链视角看，MLIR/LLVM 是 AI 编译栈中的关键第三方基础设施。
- 从代码审计视角看，项目检测对象从应用代码扩展到了编译中间表示和编译 pass 行为。
- 从安全监控视角看，系统将异常检测、聚类和溯源思想应用到编译基础设施运行日志中。

选择自由作品赛道的原因是：项目当前创新点集中在 AI 编译基础设施安全检测，而不是命题赛指定的 npm/pypi 供应链或 API/代码审计场景。自由赛道更能保留作品的技术特色和完整性。

## 5. 后续落地方向

1. 接入更多 LLVM release 和 main 分支 nightly build。
2. 增加国产 AI 芯片后端相关 dialect 和 lowering pipeline。
3. 将 MIN-004 整理为上游 issue，并验证是否仍存在于最新版本。
4. 引入结构化 IR 等价判断，降低正常优化变化造成的噪声。
5. 扩展 `mlir-runner`，从编译期 hard failure 延伸到运行时语义一致性检测。
